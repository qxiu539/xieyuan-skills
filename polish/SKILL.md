# 审校 Skill（保守模式）

## 输入处理

用户提供文章文本或文件路径。
- 如果是文件路径，先读取文件内容
- 如果是文本，直接处理

---

## 执行流程

### Pass 1：内容审校（只标注，不修改）

逐段检查以下问题，**不自动修改，只标注**：

- 数据/事实无来源 → 标注 `[⚠️  事实待核实：XXX]`
- 前后矛盾 → 标注 `[⚠️  逻辑矛盾：前文说XXX，此处说XXX]`
- 结构缺失（无引入/展开/总结）→ 标注 `[⚠️  结构问题：缺少XXX]`

Pass 1 不改原文，只在 changelog 里列出问题。

---

### Pass 2：风格审校（保守模式）

**直接修改**（明确违规，无歧义）：

1. **必删套话**（grep 全文，全部删除或替换）：
   ```
   在当今时代 / 随着科技的发展 / 在...的背景下
   值得注意的是 / 需要指出的是 / 综上所述 / 总而言之
   让我们来看看 / 首先让我们 / 接下来让我们
   在实际应用中 / 从某种意义上说 / 毋庸置疑 / 不言而喻
   赋能 / 打造 / 一站式 / 无缝 / 深度融合
   ```
   替换规则：参照替换对照表，保持自然变化，不要每次用同一个替换词。

2. **自我反应套路**（直接删除）：
   ```
   我愣了好一会儿 / 反复读了几遍 / 这句话戳到我了
   我停下来想了三秒 / 看完之后久久不能平静
   ```

3. **被动改主动**：
   - `这个问题被我们解决了` → `我们解决了这个问题`

4. **名词化改动词化**：
   - `进行了优化` → `优化了`
   - `进行操作` → `操作`

5. **书面词替换口语**（按对照表执行）：
   ```
   显著提升 → 提升了很多
   充分利用 → 用好
   有效降低 → 降低
   深入了解 → 搞清楚
   切实可行 → 靠谱
   行之有效 → 管用
   开展工作 → 开始做
   ```

6. **过度并列拆短句**：
   - `既...又...不仅...而且...同时...` → 拆成多个独立短句

**标注供用户决定**（模糊判断，不自动改）：

- `不是A而是B` 句式出现 ≥2 次 → 标注 `[💡 建议：「不是...而是」出现X次，建议保留1次，其余改为直接陈述]`
- 形容词无数据支撑（"效果很好""非常强大"）→ 标注 `[💡 建议：可替换为具体数字或场景]`
- 自我标榜式表达密集出现 → 标注 `[💡 建议：「说实话/很坦诚地讲」出现X次，建议减少]`
- 判断句/定义句（`A是B`/`所谓A即B`）→ 标注 `[💡 建议：可改为描述行为或感受]`

---

### Pass 3：细节打磨（直接修改）

1. **超长句拆短**：超过 30 字的句子拆成多个短句
2. **标点修正**：
   - 英文引号 `""` `""` → 中文引号 `「」`
   - 破折号 `——` → 逗号或句号
   - 中文后跟半角标点 `,` `:` `?` `!` → 改全角 `，` `：` `？` `！`
3. **段落间距**：确保段落之间有空行

---

### 验证步骤

对修改后的文章运行以下 grep 命令，有输出则列出行号和内容：

```bash
# 英文引号
grep -nP '[""\"'"'"']' 文章路径

# 破折号
grep -n '——' 文章路径

# 高频 AI 套话
grep -nE '在当今|随着科技|背景下|值得注意的是|需要指出的是|综上所述|总而言之|让我们|赋能|打造|一站式|无缝|深度融合|显著提升|大幅增长|充分利用|有效降低' 文章路径

# 中文后跟半角标点
grep -nP '[\x{4e00}-\x{9fa5}][,:?!]' 文章路径

# 自我反应套路
grep -nE '愣了好一会儿|反复读了几遍|戳到我了|停下来想|久久不能平静' 文章路径

# 「不是...而是」出现次数
grep -c '不是.*而是' 文章路径
```

## 输出格式

### 1. 修改后全文

直接输出完整文章，`[💡 建议：...]` 标注保留在原位。

### 2. Changelog

```
【Pass 1 · 内容问题】
- 第X段：[问题描述]（共X处）

【Pass 2 · 风格修改】
- 删除套话：XXX → XXX（共X处）
- 被动改主动：X处
- 书面词替换：X处
- 标注待决定：X处

【Pass 3 · 细节修改】
- 拆短句：X处
- 标点修正：X处

【验证结果】
- 全部通过 / 仍有X处需处理（列出具体行）
```

---

## 注意

- 保守原则：宁可少改，不要过度修改
- Pass 1 只标注不改，保护原文逻辑
- 个人态度、真实细节、个人经历——这些不替用户加，只标注缺失
- 如果用户提供的是文件路径，修改后询问是否覆盖原文件还是另存
